# Informe Tecnico - Raspberry Pi `rpi-house`

Fecha base: `2026-04-27`

## A. Estado actual

- Host: `rpi-house`
- SO: Debian/trixie sobre Raspberry Pi 5
- RAM: `7.9 GiB`, uso observado bajo
- Swap: `2 GiB`, sin uso durante la auditoria
- Disco raiz: `24%` usado
- Docker: `29.4.1`
- Swarm: activo, nodo manager leader
- Dokploy: `v0.28.8`, contenedor healthy en `:3000`
- Otros componentes activos: MicroK8s, Netdata, Prometheus, Grafana, Pi-hole, WireGuard, LightDM

## B. Problemas detectados

- El panel Dokploy por IP LAN devolvia `404` porque Traefik solo enrutaba por `Host(rpi-personal.duckdns.org)`.
- ACME/Let's Encrypt fallaba repetidamente para `rpi-personal.duckdns.org`.
- Habia secretos sensibles persistidos en Dokploy y compose raw.
- El primer intento de backup fallaba al leer volumenes vivos de PostgreSQL.
- El comando SFTP de `restic` quedo inicialmente mal parametrizado y fue corregido.

## C. Riesgos criticos

- Llaves SSH, tokens, passwords y secretos de apps estaban presentes en la base de Dokploy.
- El dominio publico del panel tenia problemas de emision TLS.
- El host ejecuta varios stacks y servicios simultaneos; hoy no hay saturacion, pero si complejidad operativa.

## D. Dokploy: causa del fallo actual

- El problema visible mas inmediato era de routing, no de caida del panel.
- Acceder por IP LAN no matcheaba ningun router Traefik de Dokploy.
- Se agrego un router LAN dedicado para `192.168.100.45`, `rpi-house`, `rpi-house.local` y `127.0.0.1`.

## E. Solucion inmediata aplicada

- Snapshot previo de configuracion.
- Instalacion de framework de backups en `/opt/backup-platform`.
- Configuracion de backup diario por cron.
- Habilitacion de `fail2ban`.
- Hardening SSH seguro sin cortar autenticacion actual por password.
- Router LAN de Dokploy agregado y validado.
- Backup real ejecutado con snapshots de:
  - `docker-volumes`
  - `dokploy-config`
  - `dokploy-db`
  - `system`
- Restore de prueba validado con snapshot `dokploy-db`.

## F. Optimizacion recomendada

- Evaluar si MicroK8s, LightDM, CUPS, Bluetooth y ModemManager siguen siendo necesarios.
- Rotar secretos expuestos y moverlos a una gestion mas segura.
- Revisar el exportador `speedtest` unhealthy.
- Definir politica de logs Docker adicional si el host sigue creciendo.

## G. Plan de backups automatico implementado

- Repositorio local del framework: `backup-platform`
- Instalacion operativa en Raspberry: `/opt/backup-platform`
- Config global: `/etc/backup/global.env`
- Definicion de origen local: `/etc/backup/servers.d/rpi-house.env`
- Retencion: 7 diarios, 4 semanales, 6 mensuales
- Destino offsite: `root@157.173.125.57:/srv/backups/rpi-house`
- Catalogo local: `/var/lib/backup/catalog/backups.sqlite3`
- Logs: `/var/log/backup`
- Programacion: `/etc/cron.d/backup-platform`
- Restore helper: `/opt/backup-platform/scripts/backup-restore-helper.sh`

## H. Proximos pasos priorizados

1. Rotar secretos expuestos en Dokploy, DuckDNS, Telegram, Pi-hole y aplicaciones.
2. Decidir que servicios locales realmente deben quedar activos en la Raspberry.
3. Agregar nuevos origenes al framework con `config/servers.d/*.env` y hooks dedicados.
4. Versionar y subir este repositorio a Git remoto.
5. Corregir la publicacion publica/TLS de `rpi-personal.duckdns.org` si el panel debe exponerse fuera de LAN.
